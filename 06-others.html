Method chaining

Le method chaining est un pattern qui permet d'appeler plusieurs méthodes sur un même objet de manière fluide en les enchaînant. Chaque méthode retourne l'objet lui-même, ce qui permet d'appeler la méthode suivante directement.

Cette méthode est également connue sous le nom de "fluent interface".

Exemple : 

class Person {
  private $name;
  private $age;

  public function setName($name) {
    $this->name = $name;
    return $this;
  }

  public function setAge($age) {
    $this->age = $age;
    return $this;
  }
}

$person = new Person()
  ->setName("John")
  ->setAge(30);

Front Controller

Le pattern Front Controller est un pattern architectural qui centralise la gestion des requêtes entrantes dans une application web. Il utilise un contrôleur unique pour traiter toutes les requêtes, ce qui permet de gérer la logique de routage, l'authentification, la gestion des erreurs et d'autres préoccupations transversales.

Ce pattern est souvent utilisé dans les frameworks MVC (Model-View-Controller) pour séparer les responsabilités et améliorer la maintenabilité du code.

Unit of Work

Le pattern Unit of Work est un pattern de conception qui gère les transactions et les modifications d'objets dans une application. Il suit les changements apportés aux objets et coordonne l'écriture de ces changements dans la base de données de manière efficace.

Ce pattern est souvent utilisé dans les systèmes de gestion de bases de données et les ORM (Object-Relational Mapping) pour minimiser le nombre d'opérations de base de données et garantir la cohérence des données.

Repository

Le pattern Repository est un pattern de conception qui abstrait l'accès aux données dans une application. Il fournit une interface pour interagir avec la source de données (base de données, API, etc.) sans exposer les détails de l'implémentation.

Ce pattern est souvent utilisé pour faciliter les tests unitaires en permettant de remplacer facilement la source de données par une implémentation fictive (mock) lors des tests.

Mock

Le pattern Mock est un pattern de conception utilisé principalement dans les tests unitaires pour simuler le comportement d'objets réels. Il permet de créer des objets factices qui imitent le comportement des objets réels, ce qui facilite le test des composants isolés sans dépendre de leurs dépendances externes.

Ce pattern est souvent utilisé pour vérifier les interactions entre les objets et s'assurer que les méthodes sont appelées avec les bons paramètres.

Data transfer object

Le pattern Data Transfer Object (DTO) est un pattern de conception qui transfère des données entre différentes couches d'une application. Il encapsule les données dans un objet simple sans logique métier, ce qui facilite le transfert des données.

Ce pattern peut par exemple être utilisé pour transférer des données entre la couche de présentation et la couche modèle d'une application.
